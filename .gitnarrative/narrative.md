## Project Overview

YSS (Youth Soccer Schedules) is a full-stack platform for ingesting, displaying, and analyzing youth soccer match data sourced from the Modular11 API. The backend is built on Azure Functions (C#/.NET) with Entity Framework Core and Azure SQL, while the frontend is a React/TypeScript SPA. An Azure Cognitive Search integration powers AI-assisted team autocomplete. The project originated as an MLSNext prototype and was rebranded to YSS during development.

The architecture separates concerns across four main C# projects: `YSS.Ingestion` (data pipeline and parsing), `YSS.Functions` (HTTP/timer-triggered Azure Functions serving the API), `YSS.Data` (EF Core entities and DbContext), and `YSS.Verification` (manual/CI ingestion runner). A nightly scheduled function recomputes ELO ratings and homepage statistics snapshots. The React frontend is a multi-page app with Schedules, Standings, Analytics, and Power Rankings pages.

## Key Features & Milestones

### Core Platform: Ingestion Pipeline & React Frontend (Feb 26 – Mar 10, 2026)

Built the foundational data pipeline using `ScheduleParser` to extract match data from Modular11's API, with a rolling 16-week ingestion window and in-memory tournament season caching to minimize unnecessary API calls. The React frontend launched with seasonal filtering, venue maps, add-to-calendar, team logo CDN integration, and a Standings page that proxies HTML parsing to Modular11 rather than computing standings from raw match data.

Key decisions:
- **Standings delegation to Modular11 proxy** — Rather than recomputing standings from raw match records, the system proxies Modular11's HTML parsing, staying synchronized with the official source and reducing complexity.
- **Rolling 16-week ingestion window** — Full historical ingestion was rejected in favor of a current-season-relevant window to control API call volume.
- FEST tournament support added alongside standard league ingestion.

### ELO Power Rankings System (Mar 16 – Mar 25, 2026)

Implemented a stateless ELO calculator with nightly database persistence (`EloCalculator`). Ratings are scoped per program (Academy vs. Homegrown) and per age group, with separate ELO histories rather than unified ratings across programs. A nightly trigger recomputes all ratings from match history and persists them; `GetPowerRankings` reads stored values rather than recomputing on request, establishing stored ELO as the system's single source of truth.

Azure Cognitive Search was integrated for team autocomplete on the Schedules page, filtering by program server-side rather than client-side, providing scalable search as the team roster grows.

## Traps & Gotchas

- **Team name trimming at both layers** — Whitespace is trimmed in both `MatchUpsertService` (ingestion) and `StandingsPage` (presentation) because upstream Modular11 data has inconsistent leading whitespace. Removing either layer will cause display mismatches or broken lookups. (`YSS.Ingestion/Services/MatchUpsertService.cs`, `YSS.Web/src/pages/StandingsPage.tsx`)

- **Stored ELO as single source of truth** — `GetPowerRankings` deliberately reads pre-stored ELO values rather than recalculating. If someone "optimizes" this by calling `EloCalculator` directly at request time, rankings will diverge from what is shown on match cards (which also use stored values). (`YSS.Functions/Triggers/GetPowerRankings.cs`, `YSS.Functions/Services/EloCalculator.cs`)

- **Program-scoped ELO intentionally separate** — Academy and Homegrown teams have distinct ELO ratings even within the same organization. This is deliberate — they compete in different contexts. Merging them into a unified rating would misrepresent team strength. (`YSS.Data/Entities/TeamAgeGroupElo.cs`)

- **DbContext queries in GetHomepageStats must remain sequential** — Parallel `Task.WhenAll` across multiple DbContext queries caused threading violations. The sequential execution pattern looks like unnecessary pessimization but is required for DbContext thread-safety. (`YSS.Functions/Triggers/GetHomepageStats.cs`)

- **FEST entries excluded from geographic region count** — FEST competition entries appear in the same data feed as geographic regions but must be explicitly filtered out of homepage region counts. The filter exists because FEST is not a geographic region; removing it inflates the displayed region metric. (`YSS.Functions/Triggers/GetHomepageStats.cs`)

- **Standings delegated to Modular11 proxy, not computed locally** — The standings figures users see are parsed from Modular11's HTML response, not derived from the locally stored match records. Any attempt to "compute standings from our data" would produce results inconsistent with the authoritative source. (`YSS.Functions/Triggers/GetStandings.cs`)

- **U19 key stored as `U19`, displayed as `U18/19`** — The database stores the Modular11 age group key as `U19`, but all display and processing logic normalizes it to `U18/19`. Normalizing at storage time or removing the translation layer will break homepage stats aggregation. (`YSS.Functions/Triggers/GetHomepageStats.cs`)

- **NJ Cup ingestion scheduled only Apr 16–21** — The `.github/workflows/njcup-ingestion.yml` workflow has hardcoded date bounds for the competition window. It will not run outside those dates and will need manual date updates for future events. (`.github/workflows/njcup-ingestion.yml`)

- **MLSNextSchedule.slnx was deleted then restored** — The solution file was removed as "cleanup" and then had to be restored as essential build infrastructure. It should not be deleted again. (`MLSNextSchedule.slnx`)
